Мета роботи
Метою роботи є дослідження та систематизація трьох основних методів рефакторингу коду — Add Parameter, Split Temporary Variable та Remove Control Flag — з урахуванням SOLID-принципів на прикладі програм мовою C++.
Завданням є демонстрація того, як рефакторинг дозволяє покращити структуру програмного забезпечення, підвищити його читабельність, знизити зв’язність і забезпечити стабільність роботи без зміни поведінки програми.

Хід роботи
Вибір та обґрунтування теми (Слайди 2–3)
Рефакторинг — це процес поліпшення структури програмного коду без зміни його функціональності.
З часом будь-яка програма стає складнішою: додаються нові функції, змінюється логіка, з’являються помилки. Без регулярного рефакторингу код поступово втрачає зрозумілість і стає важким для підтримки.
Було обрано тему методів Add Parameter, Split Temporary Variable та Remove Control Flag, оскільки вони є базовими техніками покращення коду відповідно до принципів SOLID:
• S (Single Responsibility) — кожен елемент має виконувати лише одну функцію;
• (Open/Closed) — модулі мають бути відкриті для розширення, але закриті для модифікації;
• D (Dependency Inversion) — залежності мають спрямовуватись від конкретного до абстрактного.
Рефакторинг таких типів дозволяє покращити архітектуру та підтримуваність коду в реальних C++-проєктах.
Метод 1: Add Parameter (Слайди 4–6)
Add Parameter (едд параметер) — метод, що використовується тоді, коли функція залежить від глобальних змінних або зовнішнього контексту.
До рефакторингу:
Функція використовує глобальні дані, через що стає залежною від середовища. Це ускладнює тестування, підвищує ризик помилок та робить код менш гнучким.
Після рефакторингу:
Усі необхідні дані передаються через параметри функції. Тепер вона самостійна, передбачувана, легко тестується та відповідає принципу Dependency Inversion — тобто не залежить від конкретних реалізацій.
Переваги застосування:
• Код стає більш модульним і гнучким;
• Зменшується кількість неявних залежностей;
• Функцію можна перевикористати в інших частинах програми.

Метод 2: Split Temporary Variable (Слайди 7–9)
Split Temporary Variable (спліт темпорері веріабл) — метод, який передбачає розділення однієї тимчасової змінної на кілька окремих, якщо вона використовується для різних обчислень.
До рефакторингу:
У коді використовується змінна temp, яка спочатку містить одне значення (наприклад, діаметр), потім інше (наприклад, скоригований діаметр), а в кінці використовується для обчислення довжини кола. Це створює плутанину та ускладнює налагодження.
Після рефакторингу:
Змінні отримують окремі імена — diameter, adjustedDiameter, circumference.
Така заміна робить код більш зрозумілим, а також реалізує принцип Single Responsibility — кожна змінна виконує лише одну функцію.
Переваги:
• Покращується читабельність коду;
• Зменшується ризик логічних помилок;
• Підвищується точність і передбачуваність результатів.

Метод 3: Remove Control Flag (Слайди 10–12)
Remove Control Flag (рімув контрол флег) — метод, який усуває зайві змінні-прапорці, що контролюють потік виконання програми.
До рефакторингу:
Цикл або функція містить змінну flag, яка визначає момент виходу з циклу. Це створює додатковий рівень логіки, ускладнює розуміння коду й може призвести до помилок.
Після рефакторингу:
Прапорець замінюється на прямий вихід із циклу за допомогою операторів break, return або continue.
Код стає коротшим, прозорішим і легшим у підтримці.
Переваги:
• Зниження складності логіки;
• Спрощення умовних конструкцій;
• Покращення читабельності та надійності коду.

Висновки (Слайд 13)
У результаті роботи досягнуто поставленої мети — вивчено й продемонстровано практичне застосування трьох методів рефакторингу коду на прикладі мови C++.
Отримані результати доводять, що систематичне застосування технік рефакторингу значно підвищує якість програмного забезпечення.
Основні переваги застосованих методів:
• Покращення архітектури та зменшення зв’язностей між компонентами.
• Зростання стабільності та передбачуваності роботи коду.
• Підвищення зручності тестування і розширення функціоналу.
• Відповідність принципам SOLID, що гарантує чисту архітектуру.
• Підвищення ефективності командної розробки завдяки зрозумілому стилю коду.
